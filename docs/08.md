## 一、為什麼需要物件型別

- 物件型別可以明確描述物件有哪些欄位、每個欄位是什麼型別。

## 二、如何寫物件型別？

- 可以直接在物件前標註型別。
- 當結構開始重複時，不適合一直把型別內聯寫在每個地方。
- 這時通常會改用可重複使用的型別名稱。

## 三、`type` 與 `interface`

- 兩者都可以描述物件結構。
- 巢狀物件與動態 key 也都能表達。

## 四、optional 與 readonly

- optional 用來表示欄位可以不存在。
- `| undefined` 表示欄位存在，但值可能是 `undefined`。
- readonly 用來限制欄位建立後不可修改。

## 五、`type` 的能力

- `type` 可以做聯集與交集等型別運算，適合用在需要組合、轉換、運算型別的情境。。
- 不只可描述物件，也能描述函式、字串聯集、tuple 等型別。
- `type` 是 TypeScript 裡更通用的型別工具。

## 六、`interface` 的能力

- `interface` 可以用 `extends` 延伸既有結構。
- 支援 Declaration Merging，可讓同名介面自動合併。
- 不能做 Union 和 Intersection

## 七、type 和 interface 可以互相代替使用

- 如果物件結構相同，type 和 interface 是可以互相替用的

## 八、`type` vs `interface` 怎麼選

1. TS 官方團隊建議先使用 interface
2. 依照團隊為主
3. 如果團隊還沒規範：
   a. 如果重點是描述穩定物件結構，通常可優先考慮 `interface`。
   b. 如果需要聯集、交集或更複雜的型別組合，通常用 `type`。

## 九、`as const`

- `as const` 可以保留物件裡更精準的 Literal 值。
- 同時也會讓屬性變成 `readonly`。
- 很適合搭配固定狀態、常數表、設定物件一起使用。

## 十、`enum`

- `enum` 也是 TypeScript 提供的固定選項工具。
- 舊專案中較常見，現代前端則常以 Literal Union 或 `as const` 取代。
- `enum` 在編譯後，會存留在 JS 當中。
